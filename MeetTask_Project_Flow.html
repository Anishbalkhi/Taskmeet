SECTION 01

Project Overview

MeetTask is a collaborative productivity platform that combines workspace management, task tracking, meeting scheduling, and team communication into a single unified application. It implements role-based access control, email-verified authentication, real-time notification delivery, and a fully responsive UI — making it production-ready and interview-demonstrable.

🏢

Core Features

  • Multi-workspace support per user
  • Role-based member management
  • Task creation, assignment & tracking
  • Meeting scheduling with participants
  • Workspace invitation system
  • In-app notification center
  • User profile management
🛡️

Security Features

  • JWT-based stateless authentication
  • Email verification on signup
  • Forgot/reset password via secure token
  • Password hashing with bcryptjs
  • Protected routes (frontend guard)
  • Auth middleware on all private APIs
  • Input validation (express-validator)

SECTION 06

Authentication Flow

1

Registration

User submits name, email, password. express-validator validates input. Password is hashed using bcryptjs (salt rounds = 10). User is saved to MongoDB. An email verification token is generated (crypto) and stored with expiry. Nodemailer/SendGrid sends a verification link.

2

Email Verification

User clicks the link in the email. The GET /api/auth/verify-email?token=... endpoint is called. Token is matched and expiry is checked. is_email_verified is set to true in the DB. User is redirected to the verified page.

3

Login

User submits email + password. Server fetches user (with select: '+password'). Compares using bcrypt.compare(). Checks email is verified. If valid, signs a JWT (with user ID payload) and returns it. Frontend stores the token in localStorage.

4

Authenticated Requests

axiosClient attaches the JWT as a Bearer token in every request header. The authMiddleware on the server verifies the token using jwt.verify(), extracts req.user, and passes it to the controller. Invalid/expired tokens return 401 Unauthorized.

5

Forgot / Reset Password

User requests a reset — server generates a secure token, saves it with expiry, and emails a link. User submits new password with the token. Server validates the token, hashes the new password, and saves it. Old token is cleared to prevent reuse.

⚛️

Role-Based Access Control (RBAC)

Roles are derived dynamically from the user's membership role in the current workspace. The RoleContext exposes a can("permission") helper that components use to conditionally render actions.

SECTION 10

Workspace & Notification Flow

🏢

Workspace Lifecycle

  • User creates workspace (becomes owner)
  • Owner invites members by email
  • A Notification of type workspace_invite is created
  • Invited user sees it in Notifications page
  • On Accept → added to workspace members array
  • On Decline → inviteStatus → declined
  • Multiple workspaces supported per user
📋

Task Assignment Flow

  • ADMIN or HOST creates a task in a workspace
  • Task can be assigned to any workspace member
  • A Notification of type task_assigned is created for the assignee
  • Assignee sees the new task in their dashboard
  • Task moves through: Todo → InProgress → Completed
  • Due dates + priority (Low/Medium/High) supported

SECTION 11

End-to-End User Journey

📝Register & Verify Email
🔑Login → JWT Token
🏢Create Workspace
👥Invite Members
✅Create & Assign Tasks
📅Schedule Meeting
🔔Receive Notifications
📊Track Progress
A

New User Onboarding

User sign up → verify email → land on dashboard. AuthContext stores JWT + profile. WorkspaceContext fetches workspaces (empty initially). User is prompted to create their first workspace.

B

Workspace Setup

User creates a workspace — they become the owner (ADMIN role). They can invite other users by email. Each invite triggers a notification in the invitee's account. Accepted invites add the user to the members[] array with a role.

C

Daily Team Workflow

ADMIN creates tasks and assigns them to members. Members see tasks in UserHome (personal) and TeamHome (team view). Tasks are updated through statuses. Meetings are scheduled and participants notified. All actions respect the RBAC permission model.

D

Profile & Settings

User can update their bio, role name, phone, location, and website. They can change their password (current password required). Theme (dark/light) preference is persisted locally. Workspace selection persists across sessions via localStorage.
